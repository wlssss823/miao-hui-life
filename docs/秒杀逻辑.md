# 秒杀逻辑详解

## 整体数据流

```
用户请求 seckillVoucher(voucherId)
         │
         ▼
┌─────────────────────────┐
│  ① IP 防刷              │  Lua 滑动窗口，每 IP 每分钟 10 次
│  是否超出限频？          │
└─────────┬───────────────┘
          │ 通过
          ▼
┌─────────────────────────┐
│  ② 令牌桶限流            │  Redisson RRateLimiter 200/s
│  是否超出阈值？          │
└─────────┬───────────────┘
          │ 通过
          ▼
┌─────────────────────────┐
│  ③ 时间校验              │  Java 层面
│  是否在秒杀时间段内？    │
└─────────┬───────────────┘
          │ 通过
          ▼
┌─────────────────────────┐
│  ④ Lua 原子脚本 (Redis)  │  一次网络 IO 完成：
│  ─ 检查库存             │  库存判断 + 幂等判断 + 扣库存 + 记录用户
│  ─ 检查重复下单         │
│  ─ 扣库存               │
│  ─ 记录用户 Set         │
└─────────┬───────────────┘
          │ r=0 通过
          ▼
┌─────────────────────────┐
│  ⑤ 发送 MQ 消息          │  异步削峰，直接返回前端成功
│     order_queue          │
└─────────┬───────────────┘
          │ 异步消费
          ▼
┌─────────────────────────┐
│  ⑥ @RabbitListener      │  消费端，并发 4
│  ─ Redisson 锁           │  一人一单（分布式锁）
│  ─ createVoucherOrder()  │  事务落库 + 销量统计 + 延迟取消
└─────────┬───────────────┘
          │
          ▼
      订单落库成功
          │
          ▼
┌─────────────────────────┐
│  ⑦ 超时未支付自动取消     │  死信队列 30 分钟 TTL
│  ─ 恢复 Redis 库存       │
│  ─ 移除下单用户记录      │
│  ─ 发送取消通知          │
└─────────────────────────┘
```

## 三层幂等（防重复下单）

| 层级 | 方式 | 说明 |
|------|------|------|
| 第一层 | Lua 脚本 Redis Set | 同用户同优惠券，Set 中已存在则返回 2 |
| 第二层 | Redisson 分布式锁 | 消费端 `tryLock`，锁 key `lock:order:{userId}` |
| 第三层 | DB 查询 | `createVoucherOrder` 中 `query().eq("user_id", userId).eq("voucher_id", voucherId).count()` |

即使某一层失效，后续层级仍能兜底。

## Redis 使用

### 秒杀相关 Key

| Key | 类型 | 用途 |
|-----|------|------|
| `seckill:stock:{voucherId}` | String | 优惠券库存数量 |
| `seckill:order:{voucherId}` | Set | 已下单用户 ID 集合（幂等判断） |

### Lua 脚本（VerifyRedis.lua）

```lua
-- 入参：voucherId, userId, orderId
-- 返回：0=正常 1=库存不足 2=重复下单

local stockKey = 'seckill:stock:' .. voucherId
local orderKey = 'seckill:order:' .. voucherId

-- 1. 检查库存
if(tonumber(redis.call('get', stockKey)) <= 0) then return 1 end

-- 2. 检查重复下单
if(tonumber(redis.call('sismember', orderKey, userId)) == 1) then return 2 end

-- 3. 扣库存 + 记录用户
redis.call('incrby', stockKey, -1)
redis.call('sadd', orderKey, userId)

return 0
```

三条 Redis 命令通过 Lua 脚本封装为一次网络 IO + 原子执行。

### Redisson 分布式锁

消费端 `handleVoucherOrder` 中：

```java
RLock lock = redissonClient.getLock("lock:order:" + userId);
boolean isLock = lock.tryLock();
```

- 锁 Key：`lock:order:{userId}`（按用户粒度，不影响其他用户）
- 加锁失败直接返回，不阻塞
- 解决的是高并发下同一用户多次请求到达消费端的并发问题

## 是否达标判断（完整校验链）

| 步骤 | 校验内容 | 通过条件 | 不通过返回 |
|------|----------|----------|-----------|
| ① IP 防刷 | `rate:ip:{ip}` 计数器 ≤ 10 | 未超限频 | `"操作太频繁，请稍后再试"` |
| ② 令牌桶限流 | RRateLimiter `tryAcquire` | 获取到令牌 | `"秒杀活动太火爆，请稍后再试"` |
| ③ Java 时间判断 | `beginTime ≤ now ≤ endTime` | 在时间段内 | `"不在秒杀卷购买时间区间中"` |
| ④ Lua 库存检查 | `stock > 0` | 有库存 | `"库存不足"` |
| ⑤ Lua 重复下单检查 | Set 中无此 userId | 首次下单 | `"不能重复下单"` |
| ⑥ 消费端 Redisson 锁 | 同一 userId 未持有锁 | 未持有 | `"不允许重复下单"` |
| ⑦ DB 一人一单查询 | 表内无此 userId+voucherId 记录 | 无记录 | `"订单已存在"` |
| ⑧ DB 乐观锁扣库存 | `stock > 0` 条件更新 | 更新行数 > 0 | `"库存不足"` |

## @Transactional 事务方法（createVoucherOrder）

```java
@Transactional
public void createVoucherOrder(VoucherOrder voucherOrder) {
    // 1. DB 查重（一人一单）
    int count = query().eq("user_id", userId).eq("voucher_id", voucherId).count();
    if (count > 0) return;

    // 2. 扣库存（乐观锁条件：stock > 0）
    seckillVoucherService.update()
            .setSql("stock = stock - 1")
            .eq("voucher_id", voucherId)
            .gt("stock", 0)
            .update();

    // 3. 保存订单
    save(voucherOrder);
}
```

注意点：
- 类内部调用事务方法会失效，所以消费端用 `AopContext.currentProxy()` 获取代理对象再调用
- `gt("stock", 0)` 是乐观锁 CAS，防止超卖
- `save(voucherOrder)` 中的 `voucherOrder` 已在 Lua 阶段生成了全局唯一 ID（通过 RedisIdGenerator）

## 为什么用 MQ 削峰

如果不加 MQ，高并发下所有请求都直接打 DB：
- DB 连接池被打满 → 接口超时 → 前端重试 → 雪崩

引入 MQ 后：
1. Lua 脚本（纯内存）挡住大部分流量
2. 通过的消息扔到队列，秒回前端成功
3. 消费端单线程消费，DB 压力恒定

这就是**异步下单 + 流量削峰**的核心思路。
