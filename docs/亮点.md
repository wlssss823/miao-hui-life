# 项目亮点

## 1. 缓存架构：CacheClient 三种模式

通用缓存工具类，解决缓存三大问题：

- **PassThrough（缓存穿透）** — 查询未命中时缓存空值（TTL 2 分钟），防止恶意 key 穿透到 DB
- **LogicalExpire（缓存击穿）** — 缓存中存逻辑过期时间，过期后只允许一个线程查 DB 重建缓存，其他线程继续返回旧数据。适用于热点 key
- **Mutex（缓存击穿）** — 互斥锁自旋等待，谁拿到锁谁查 DB 重建，其他线程自旋重试

```java
// 调用方一行切换策略
cacheClient.queryWithPassThrough(key, id, type, dbFallback, ttl, unit);
cacheClient.queryWithLogicalExpire(key, id, type, dbFallback, ttl, unit);
cacheClient.queryWithMutex(key, id, type, dbFallback, ttl, unit);
```

现已重构为**策略模式（Strategy Pattern）**：

```
┌─────────────────┐     ┌──────────────────────┐
│  CacheClient    │────▶│  CacheStrategy       │
│  (Context)       │     │  (接口)               │
│                  │     ├──────────────────────┤
│  query("passThrough"...)
│  query("logicalExpire"...)   PassThroughCacheStrategy
│  query("mutex"...)           LogicalExpireCacheStrategy
└─────────────────┘           MutexCacheStrategy
```

```java
// 调用方只需指定策略名
shop = cacheClient.query("logicalExpire", CACHE_SHOP_KEY, id, Shop.class, this::getById, CACHE_SHOP_TTL, TimeUnit.MINUTES);
```

三个策略实现互相独立，遵循开闭原则：新增缓存策略只需新建一个 `CacheStrategy` 实现类，无需修改 `CacheClient`。
```

## 2. 分布式全局 ID 生成器（RedisIdGenerator）

```java
// 41 bit 时间戳（秒） | 23 bit 序列号
// 序列号按天自增，key = icr:{prefix}:{yyyy:MM:dd}
// 每天一个 key，自动重置
long id = timestamp << 32 | count;
```

- 64 位 long，全局唯一、趋势递增
- 按天分 key，避免单 key 递增瓶颈
- 比数据库自增 ID 更灵活（不依赖 DB）

## 3. Token 双拦截器 + 自动续期

登录信息存 Redis Hash，双拦截器协同：

| 拦截器 | 优先级 | 职责 |
|--------|--------|------|
| `RefreshTokenInterceptor` | 先执行（order 0） | 所有请求：解析 token → 查 Redis → 放到 ThreadLocal → **续期 TTL** |
| `LoginInterceptor` | 后执行（order 1） | 只拦截需要登录的路径：检查 ThreadLocal 是否有用户 |

效果：只要用户还在操作，token 就自动续期 60 分钟；不强制登录的页面也能判断"当前是谁"。

## 4. 附近商铺 GEO 查询

用 Redis GEO 数据结构实现：

```java
// 存入店铺坐标
GEOADD shop:geo:{typeId} longitude latitude shopId

// 按距离查询 + 分页
GEOSEARCH key BYLONLAT x y BYRADIUS 5000 WITHDISTANCE
```

- 前端传坐标，后端按距离排序 + 分页返回
- 不需要引入 Elasticsearch 等重型中间件

## 5. ZSet 点赞 + 点赞榜单

```java
// 用 ZSet 替换 Set，score 存时间戳
zadd blog:liked:{blogId} timestamp userId

// 查询 Top 5 点赞用户（点赞榜单）
zrange blog:liked:{blogId} 0 4
```

- ZSet 天然支持排序，比 Set 多一个"榜单"能力
- `score` 判空判断是否已点赞，避免重复点赞

## 6. BitMap 签到

```java
// 一个月一个 key，某一天签到就把对应 bit 置 1
key = sign:{userId}:{yyyyMM}
setbit key day-1 1

// 统计连续签到天数：取出当月所有 bit，从右往左数连续 1 的个数
bitfield key get u{dayOfMonth} 0
```

32 个用户一个月的签到只需 4 字节，极致省内存。

## 7. 写扩散推流 + 滚动分页

发布博客时，推送到所有粉丝的信箱（ZSet），粉丝刷新时从自己信箱拉取：

```
发博客 → 遍历粉丝 → zadd feed:{fansId} timestamp blogId
看动态 → ZREVRANGEBYSCORE feed:{myId} max min LIMIT offset 2
```

- **写扩散**：发一次，推给所有粉丝。适合粉丝量不大的场景
- **滚动分页**：用 score（时间戳）+ offset 实现无游标分页，处理同一时刻多发 blog 的重复问题

## 8. Set 共同关注

```java
// 每人一个 Set，存关注的人
sadd follow:{userId} followUserId

// 共同关注 → Set 交集
sinter follow:{curUserId} follow:{aimUserId}
```

SINTER 命令在 Redis 内完成计算，不需要拉取到应用层做交集。

## 9. 可靠消息投递（RabbitMQ Confirm）

```java
// 发送时绑定 CorrelationData（含订单 ID）
rabbitTemplate.setConfirmCallback(...);   // 确认是否到达 Exchange
rabbitTemplate.setReturnsCallback(...);   // 确认是否路由到 Queue
```

生产端双重确认，防止消息丢失。

## 10. 乐观锁扣库存（CAS）

```java
// 扣减库存时带上 gt("stock", 0) 条件
seckillVoucherService.update()
    .setSql("stock = stock - 1")
    .eq("voucher_id", voucherId)
    .gt("stock", 0)     // CAS 乐观锁，防止超卖
    .update();
```

不用悲观锁，并发度高；`gt("stock", 0)` 替代 version 字段，更简洁。

## 11. 三层幂等（防重复下单）

| 层级 | 方式 | 存储 | 说明 |
|------|------|------|------|
| 第一层 | Lua 脚本 Redis Set | 内存 | 秒杀请求入口，原子检查 |
| 第二层 | Redisson 分布式锁 | 内存 | MQ 消费端，按用户粒度加锁 |
| 第三层 | DB 唯一约束查询 | 磁盘 | 事务内 COUNT 兜底 |

逐层兜底，任何一层失效仍能保证业务幂等。

## 12. IP 防刷 + 全局限流

- **IP 维度**：Lua 脚本实现滑动窗口计数器，每分钟每 IP 最多 10 次秒杀请求
- **全局维度**：Redisson RRateLimiter 令牌桶，每秒 200 个请求
- 双重限流防止恶意刷单和流量突刺

## 13. 链路追踪 traceId

- `OncePerRequestFilter` 最高优先级拦截所有请求
- 自动生成 16 位 traceId（支持上游透传 `X-Trace-Id` 头）
- 写入 SLF4J MDC，所有日志自动携带 traceId
- 响应头 `X-Trace-Id` 返回给调用方，便于定位问题

## 14. SSE 消息中心

- 基于 Spring SseEmitter 实现实时推送
- ConcurrentHashMap 管理用户连接，支持多端登录
- 整合订单/评价/点赞/关注等业务场景

## 15. 商家数据看板

- 实时 PV（String 计数器）+ UV（HyperLogLog 去重）
- 订单统计（总量、已支付、已核销、营收估算）
- 评价统计（总量、平均评分）
- 销量排行（ZSet 按日聚合）
